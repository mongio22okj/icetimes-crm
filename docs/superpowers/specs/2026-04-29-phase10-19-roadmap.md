# Phases 10–19 — Beat-Premium Roadmap

**Date:** 2026-04-29
**Status:** Draft
**Scope:** Ten phases that close the surface-area gap with premium dashboards (Metronic, TailAdmin Pro, Material Tailwind, Mosaic, Velzon, Notus, shadcn-admin, Horizon UI) while doubling down on what Django uniquely enables.

## Context

Phases 1–9 built a real backend — auth, 2FA, 20 apps, i18n, density system, OKLCh tokens, theme-aware charts, E2E tests. Architecturally Apex already beats most paid kits. The remaining gap is **documented surface area**: not enough components, no real datatable, no realtime, no API, no orgs/RBAC. This roadmap closes that gap in ten focused phases.

## Operating principles

1. **One phase = one branch = one PR**, parent branch is the previous phase.
2. **Demo-data first.** Every feature seeds via `seed_demo` so screenshots always work.
3. **HTMX over JS** wherever the round-trip cost is fine. Reach for Alpine when state is local; reach for Channels only in Phase 14.
4. **No CDN deps for new code.** New JS goes through the local bundle path established in Phase 19.
5. **Add screenshots to `screenshots/`** as part of the phase PR — README is generated from them.
6. **Add at least one E2E happy-path test per new top-level surface.**

## Phase 10 — Component library

**Goal.** A `/components/` route with a left rail of every primitive and a documented variant grid for each. This is what reviewers scroll first.

**Apps.** New `apps/components/`. Pure presentation, no models.

**Surfaces (one page per primitive).** Modal, drawer/off-canvas, toast, tooltip, popover, tabs (underline / pill / vertical), accordion, stepper, datepicker, daterange, timepicker, color picker, multi-select, tag input, combobox, file uploader (drag-drop + progress), rating, slider, toggle group, segmented control, skeleton loader, spinner, progress ring, empty-state variants (×6), avatar variants, badge variants.

**Pattern.** Each primitive is a partial under `templates/components/<name>.html`; each demo page composes 4–8 variants and shows the markup in a `<pre>` block via the existing `_codeblock.html`. Toast lives in `apps/core/messages.py` and renders Django messages framework via Alpine.

**Out of scope.** WYSIWYG editor (Phase 12), chart components (already in Phase 8 showcase).

**Acceptance.** Sidebar entry under "Showcase → Components" with sub-routes, screenshots for each primitive, E2E smoke test that opens 5 primitives and confirms render.

## Phase 11 — HTMX datatable

**Goal.** A reusable `<table>` partial backed by a `TableView` mixin that any list view can opt into. Replaces the static `pages/datatable.html` with a real one and upgrades Customers, Orders, Invoices, Products, Users.

**New module.** `apps/core/tables.py` — `TableView`, `TableConfig`, `Column` dataclass.

**Features.**
- Server-side pagination, sort, filter via HTMX partial swaps (no full reload).
- Column visibility toggle (persisted per-user in `UserPreference` JSON field).
- Row selection + bulk actions toolbar (delete, export, status change — view-defined).
- CSV export (always) + PDF export via WeasyPrint reuse + XLSX via `openpyxl`.
- Saved views per user (filter+sort combos saved with a name).
- Sticky header, optional sticky first column, responsive collapse to cards under `md`.
- Keyboard nav: ↑↓ moves row focus, Space selects, Enter opens.

**Wired into.** Customers list, Orders list, Invoices list, Products list, Users list, Activity log.

**Out of scope.** Inline editing (consider Phase 12 follow-up), column reorder via drag (nice-to-have, defer).

**Acceptance.** Lighthouse perf ≥ 95 on Customers list with 1000 seeded rows, E2E test for sort + filter + bulk delete + CSV export.

## Phase 12 — Forms 2.0

**Goal.** Replace the current forms gallery with a polished form system that matches Material Tailwind / Velzon visual quality.

**New widgets** (in `apps/core/widgets.py`).
- `FloatingLabelInput`, `FloatingLabelTextarea`
- `IconPrefixInput`, `IconSuffixInput`
- `MultiSelect` (Tom-Select-style, keyboard accessible, no jQuery — vanilla)
- `TagInput` (free-form tags with paste-to-split)
- `Combobox` (typeahead from URL endpoint)
- `RichText` (TipTap or EasyMDE — pick one, self-hosted)
- `FileDropzone` (drag-drop, multi-file, progress bars, image previews)
- `DateRangePicker` (lightweight vanilla, not flatpickr)
- `CharacterCounter` mixin for text fields
- `ConditionalReveal` (Alpine helper to show/hide based on another field's value)

**Apply to.** Customer create/edit, Invoice create, Profile settings, Help article create.

**Forms gallery rewrite.** `templates/pages/forms_gallery.html` becomes the canonical reference: every widget rendered with code snippet + accessibility notes + validation states (default, focus, success, warning, error, disabled, readonly).

**Out of scope.** Multi-step wizard rewrite (already shipped in Phase 8), survey/poll builder, drag-to-reorder form builder.

**Acceptance.** All new widgets pass keyboard-only navigation, E2E test creating an invoice using new widgets end-to-end, axe-core a11y check on forms gallery returns 0 violations.

## Phase 13 — Notification center

**Goal.** Promote notifications from "bell + dropdown" to a real notification system with categories, preferences, and browser push.

**Models.**
- Extend `Notification` with `category` (system, billing, mention, comment, security), `actor` FK, `target_url`.
- New `NotificationPreference(user, category, email, in_app, push)`.
- New `PushSubscription(user, endpoint, p256dh, auth)` for Web Push.

**Surfaces.**
- `/notifications/` — full-page center: filter by category, mark all read, archive, group by day.
- `/notifications/preferences/` — toggle grid: category × channel.
- Bell dropdown becomes a slim "latest 5 + view all" preview.
- Browser push opt-in via service worker (lays groundwork reused in Phase 19 PWA).
- Sound toggle (one ping for new mentions, settable in preferences).

**Backend.** `apps/notifications/dispatch.py` central dispatcher: `notify(user, category, title, body, target_url)`. Honors preferences. In-app via Django messages + DB row; email via current backend; push via `pywebpush`.

**Triggers wired in.** New invoice, payment failed, comment on assigned task, @mention in chat, new follower in profiles app.

**Out of scope.** Mobile push (APNS/FCM), SMS notifications, digest emails (Phase 17).

**Acceptance.** E2E test that triggers a notification, verifies bell badge increments, opens center, marks read; preference page persists across reload.

## Phase 14 — Realtime via Django Channels

**Goal.** Add a thin realtime layer that powers chat presence, kanban broadcast, live notification toasts, and a "live visitors" demo. This is the headline differentiator vs. static kits.

**Stack.** Django Channels 4 + Daphne (or Uvicorn) + Redis as channel layer. Dev-time fallback to in-memory layer.

**Consumers.**
- `ChatConsumer` — typing indicator, online presence, message broadcast (replaces current chat polling).
- `KanbanConsumer` — broadcasts card moves; other open boards animate to new state.
- `NotificationConsumer` — pushes new notifications to bell without polling.
- `PresenceConsumer` — global presence channel; powers "online users" pill in the header.

**Demo page.** `/realtime/` — open in two tabs, click around, see updates fire across tabs. Live visitor counter. Live cursor positions on a shared canvas (optional, time-permitting).

**Infra notes.**
- Add `redis` service to `docker-compose.yml`.
- Document scaling notes in deploy recipes (Phase 19): single-process gunicorn → daphne → multi-process behind nginx.
- Settings split: `CHANNEL_LAYERS` defaults to in-memory; prod requires `REDIS_URL`.

**Out of scope.** Voice/video (WebRTC), federated presence across orgs, message history search (defer to Phase 15 search endpoints).

**Acceptance.** E2E test that opens two browser contexts, sends chat from one, asserts arrival in the other within 2s; kanban move likewise.

## Phase 15 — API layer (Django Ninja)

**Goal.** Production-quality REST API over the existing models, with OpenAPI docs that replace the current static `/pages/api-docs/` page.

**Library.** Django Ninja (auto OpenAPI, pydantic-style schemas, lighter than DRF, async-friendly).

**Endpoints (CRUD + filters).** Customers, Orders, OrderItems, Products, Invoices, InvoiceItems, Files/Folders, Notifications, ChatMessages, KanbanCards, Events, Projects/Tasks.

**Auth.** Token-based (`APIKey` model: name, key hash, last_used, scopes). UI for managing keys lives in Phase 17 settings.

**Features.**
- Cursor pagination + filter + ordering.
- Rate limit headers (`X-RateLimit-*`) via `django-ratelimit`.
- Idempotency keys on POST endpoints.
- Webhooks: `Webhook(user, url, secret, event_types[])` with retries + signed payloads.

**Surfaces.**
- `/api/v1/...` mounted in `apex/urls.py`.
- `/api/docs/` — Swagger UI.
- `/api/redoc/` — Redoc.
- `/settings/api-tokens/` — list/create/revoke (Phase 17 wires this).

**Out of scope.** GraphQL, gRPC, full event-sourcing.

**Acceptance.** All endpoints have request/response schemas + at least one example, tests cover 200/401/403/404/429 paths for representative endpoints, OpenAPI spec validates with `openapi-spec-validator`.

## Phase 16 — Organizations + RBAC

**Goal.** Multi-tenant primitives so the dashboard reads as a B2B SaaS demo, not a single-user toy.

**Models.**
- `Organization(name, slug, plan_fk, created_by)`.
- `Membership(user, organization, role)` — `owner / admin / member / billing / viewer`.
- `Invitation(organization, email, role, token, expires_at, accepted_at)`.
- `Role` + `Permission` matrix (resource × action) seeded for default roles, overridable per-org.

**Surfaces.**
- Org switcher in user menu (top-left or top-right per design call).
- `/organizations/<slug>/settings/` — name, logo, plan, members, invitations, roles.
- `/organizations/<slug>/members/` — list, invite, change role, remove.
- `/organizations/<slug>/roles/` — checkbox grid for permissions.
- Invitation accept/decline flow at `/invitations/<token>/`.

**Mixin.** `OrgScopedMixin` filters all org-scoped querysets by `request.organization`. `HasPermissionMixin("invoices.create")` enforces RBAC.

**Migration.** Org-scope existing models (Customer, Invoice, Order, Product, Project, KanbanCard, etc.) by adding nullable `organization` FK + backfill to a "Default" org for existing demo data.

**Out of scope.** Per-org branding/whitelabel CSS (Phase 18 nice-to-have), SCIM provisioning, SSO/SAML (separate phase if pursued).

**Acceptance.** Two seeded orgs in demo data; switching shows different customers/invoices; invite flow works end-to-end; RBAC denies a "viewer" from creating an invoice.

## Phase 17 — Settings depth

**Goal.** Bring `/settings/` from "profile + password + 2FA + appearance" up to enterprise-tier completeness.

**New settings panes.**
- **Sessions** — list active sessions (device, IP, last seen), "sign out other sessions". Uses Django's `session_key` + a `SessionMetadata` model populated by a middleware.
- **API tokens** — list/create/revoke (UI for Phase 15 keys).
- **Webhooks** — list/create/test (Phase 15 webhooks).
- **Connected accounts** — OAuth connect/disconnect for Google, GitHub (django-allauth, optional install).
- **Audit log** — per-user action history (sign-ins, security changes, data exports).
- **Data export** — request a ZIP of user's data (GDPR-shaped JSON dump). Async job stub via simple management command, link emailed when ready.
- **Account deletion** — typed-confirm flow, soft delete with 30-day grace, scheduled hard delete via management command.
- **Notification preferences** — already shipped in Phase 13, lift link into settings nav.
- **Email digest** — daily/weekly/never selection.

**Org-level settings (lives under org switcher, see Phase 16).** Members, roles, billing, audit, danger zone.

**Out of scope.** SSO/SAML, custom domains, IP allowlisting (could be a future Phase).

**Acceptance.** E2E test for each new pane (open, perform an action, verify state); audit log shows every security-relevant action from previous tests.

## Phase 18 — Marketing polish

**Goal.** Convert the marketing site from "good enough" to "this is the kit I want to buy."

**New surfaces.**
- `/changelog/` — renders `CHANGELOG.md` (or a `Release` model if we want admin-curated entries) with per-release anchors and Atom feed.
- `/roadmap/` — public Now / Next / Later board, sourced from a `RoadmapItem` model so it's editable in admin without redeploy.
- `/compare/` — comparison table: Apex vs. building from scratch / vs. competitor categories. Animated check-marks, anchor scroll between rows.
- `/customers/` — case studies (logo wall + 3–6 written stories with metrics; uses existing blog model with a `kind` field).
- Live in-page demo on `/` hero — small iframe of the dashboard with auto-rotating routes (10s per route).
- `/showcase/` — links to all dashboard variants, components, charts, forms, datatable demo. One destination for "show me everything."

**SEO basics (apply across marketing routes).** `og:` tags, Twitter cards, JSON-LD (`SoftwareApplication`, `Article`, `BreadcrumbList`), `sitemap.xml` via `django.contrib.sitemaps`, `robots.txt`.

**Existing pages tightened.** Hero variants per industry refreshed, testimonials with avatars + roles, pricing table with annual/monthly toggle and FAQ accordion below.

**Out of scope.** A/B testing harness, analytics integration beyond a configurable GA4/Plausible snippet, blog comments.

**Acceptance.** Lighthouse SEO ≥ 100 on all marketing routes, sitemap.xml validates, og:image renders correctly when the URL is shared on Slack/X.

## Phase 19 — Docs site + deploy + PWA + bundling

**Goal.** Wrap the package: hosted docs, one-click deploys, installable PWA, real frontend bundle.

**Docs site.** MkDocs Material under `docs/site/`. Hosted at `docs.colorlib.com/apex/` (or subdomain). Sections: Getting started, Project structure, Recipes (add a page, add a chart, swap the palette, add a model, write a CBV with mixins), Components reference, Deploy guides, FAQ, Migration notes per release.

**Deploy recipes.** `deploy/` directory with one config per provider:
- `render.yaml` (Render web + worker + redis + postgres).
- `fly.toml` (Fly.io).
- `railway.json` (Railway).
- `app.yaml` (DigitalOcean App Platform).
- `Dockerfile.production` + `docker-compose.prod.yml` for VPS.
- `.github/workflows/deploy-*.yml` examples per provider.

Each recipe documented in the docs site with screenshots.

**PWA.** `manifest.webmanifest`, icons (192/512/maskable), service worker (Workbox via npm) with offline shell, installability check, splash screens. Wires into the Phase 13 push subscription so a single SW handles both push and offline.

**Frontend bundling.** Replace CDN script tags + ad-hoc `static/js/*.js` with esbuild. Inputs: `static_src/js/{shell,charts,app,realtime,table,forms,components}.ts`. Outputs hashed into `static/js/dist/` so WhiteNoise's `CompressedManifestStaticFilesStorage` can cache-bust. Self-host Alpine, HTMX, ApexCharts, TipTap, Workbox via npm.

**Quality gates.**
- `django-debug-toolbar` in dev settings.
- `/__health/` (DB ping, channel-layer ping, cache ping) and `/__metrics/` (Prometheus format) endpoints.
- `pre-commit` config: ruff, prettier (templates), djade (template auto-format).
- GitHub Actions: lint + test + e2e + build + deploy-docs.

**Out of scope.** Mobile native shells, Electron desktop wrapper.

**Acceptance.** Lighthouse PWA ✓ on dashboard route, fresh `git clone && make deploy-render` works, docs site builds and deploys via GH Actions, all three new endpoints (`/__health/`, `/__metrics/`, `/api/v1/`) return expected schemas.

## Cross-phase backlog (pick into phases as fits)

These don't need their own phase but should land somewhere along the way:

- **Image optimization.** `pillow_avif_plugin` + post-save signal that emits AVIF/WebP next to original — fits Phase 12 (file uploader) or Phase 19.
- **Demo data quality.** Faker locale tuning, DiceBear avatars, realistic revenue curves — fits Phase 11 when seeding 1000-row tables anyway.
- **Search backend.** Postgres FTS via `django.contrib.postgres.search`, results page with highlighting + per-app grouping — fits Phase 15 (treat as a search endpoint).
- **Accessibility audit page.** `/components/accessibility/` documenting our patterns — fits Phase 10.
- **Reduced-motion respect** on existing animations — fits Phase 10.
- **Skip-to-content link + aria-current on nav** — fits Phase 10.
- **Self-hosted icon font fallback** for Lucide — only if perf measurements show inline SVG hurting; defer.

## What we're explicitly *not* shipping

- React/Vue rewrite of any surface. Server-rendered + HTMX is the position.
- Native mobile apps.
- ML/AI features (no auto-categorization, no chatbot embed).
- Crypto / Web3 anything.
- Visual page builder / drag-drop dashboard editor (different product).
- Multi-currency invoicing beyond display formatting.

## Execution rhythm

- One phase per branch, parent = previous phase branch.
- Each phase opens with a focused spec doc in `docs/superpowers/specs/` (this roadmap doc points at them as they land).
- Each phase closes with: tests green, screenshots refreshed, README/CHANGELOG updated, R2 publish per global instructions.
- After Phase 19 the kit should be demonstrably ahead of every premium dashboard kit on backend depth, with surface-area parity on the visible features buyers test.
