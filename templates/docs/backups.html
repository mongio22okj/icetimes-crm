The script

Save as /usr/local/bin/apex-django-backup.sh:

{% code lang="bash" %}#!/usr/bin/env bash # Daily backup of the apex_django Postgres DB with 14-day rotation. set -euo pipefail BACKUP_DIR="/var/backups/apex-django" KEEP_DAYS=14 DB_NAME="apex_django" DB_USER="apex_django" TIMESTAMP=$(date -u +%Y%m%dT%H%M%SZ) OUT="$BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.sql.gz" mkdir -p "$BACKUP_DIR" # Read DB password from the same env file the app uses. PGPASSWORD=$(grep "^DATABASE_URL=" /etc/apex-django.env \ | sed -E "s|.*://[^:]+:([^@]+)@.*|\1|") export PGPASSWORD pg_dump -U "$DB_USER" -h 127.0.0.1 "$DB_NAME" | gzip -9 > "$OUT" chmod 0600 "$OUT" # Rotate: delete dumps older than KEEP_DAYS. find "$BACKUP_DIR" -name "${DB_NAME}_*.sql.gz" -type f \ -mtime "+${KEEP_DAYS}" -delete echo "$(date -u +%FT%TZ) — backup written to $OUT ($(du -h "$OUT" | cut -f1))"{% endcode %} {% code lang="bash" %}sudo chmod +x /usr/local/bin/apex-django-backup.sh sudo install -d -o root -g root -m 0700 /var/backups/apex-django{% endcode %} {% callout type="warn" %} mode 0700 on the backup directory is critical — these dumps contain all user data and password hashes. {% endcallout %}

Cron — daily at 03:30 UTC

{% code lang="bash" %}sudo crontab -e{% endcode %}

Add:

{% code lang="cron" %}30 3 * * * /usr/local/bin/apex-django-backup.sh >> /var/log/apex-django/backup.log 2>&1{% endcode %}

Run as root since the script reads /etc/apex-django.env (mode 0640 owned by apex:apex — root has implicit access).

Restoring

{% code lang="bash" %}# Decompress and restore — make sure the DB is empty / dropped first. gunzip -c /var/backups/apex-django/apex_django_20260503T033000Z.sql.gz \ | sudo -u postgres psql apex_django{% endcode %}

For point-in-time tests, restore into a throwaway DB:

{% code lang="bash" %}sudo -u postgres createdb apex_django_test gunzip -c /var/backups/apex-django/apex_django_*.sql.gz \ | sudo -u postgres psql apex_django_test{% endcode %}

Off-host backups (recommended)

Local backups protect against DROP TABLE-by-the-app accidents but not against the whole server going away. Two cheap options:

Cloudflare R2 (S3-compatible)

Add to the backup script after the pg_dump:

{% code lang="bash" %}rclone copy "$OUT" r2pro:colorlib-backups/apex-django/ --progress{% endcode %}

restic to Backblaze B2

{% code lang="bash" %}# Initial setup restic -r b2:bucket-name:apex-django init # In the backup script, after pg_dump: restic -r b2:bucket-name:apex-django backup "$OUT" --tag postgres restic -r b2:bucket-name:apex-django forget --keep-daily 14 --prune{% endcode %}

restic deduplicates across snapshots, so 14 daily dumps of a slowly-changing DB take far less than 14× the space.

What to actually back up

For a demo, only the DB matters — everything else (/var/www/apex-django/ code + staticfiles/) is regenerable from git pull && deploy.sh. For a real product, also include:

  • /var/www/apex-django/user_files/ — uploaded files
  • /etc/apex-django.env — env file with secrets
  • /etc/nginx/sites-available/apex-django.*.conf — vhost config
  • /etc/systemd/system/apex-django.service — Daphne unit
