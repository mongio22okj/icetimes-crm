The /__health/ endpoint

{% code lang="bash" %}curl https://apex-django.example.com/__health/{% endcode %} {% code lang="json" %}{ "status": "ok", "checks": { "db": {"ok": true, "ms": 1.2}, "channel_layer": {"ok": true, "ms": 0.4} }, "failed": [] }{% endcode %}

Returns:

  • 200 when every required check passes
  • 503 when any check fails (with {"status": "degraded", "failed": [...]})

The endpoint is anonymous (no auth) and no-cache (no edge caching). Wired in apex/urls.py at /__health/ (double underscore = "internal, not for humans").

What it checks

CheckWhat it doesWhy
db SELECT 1 confirms Postgres is reachable + connection pool is alive
channel_layer sends + receives one message through the layer confirms Redis (in prod) or in-memory layer is functional

Each check is wrapped in try/except so a probe failure never raises 500 — the response always returns valid JSON with the failure mode in the body.

Wiring an uptime monitor

UptimeRobot (free tier)

  1. Add a monitor of type HTTP(S).
  2. URL: https://apex-django.example.com/__health/
  3. Keyword monitoring: "status": "ok" — alerts when the keyword disappears.
  4. Interval: 5 min.

Cron-based self-check (zero external deps)

{% code lang="cron" %}*/5 * * * * curl -fsS https://apex-django.example.com/__health/ | grep -q '"status": "ok"' || systemctl restart apex-django{% endcode %} {% callout type="warn" %} Auto-restart loops can spiral if the underlying problem is e.g. an out-of-memory condition. Wrap in something smarter (systemd RestartSec= + restart limit) for production. {% endcallout %}

Sentry — error capture (opt-in)

Apex has Sentry SDK pre-wired (gated on SENTRY_DSN env var). To enable:

{% code lang="bash" %}# 1. Install the optional dep sudo -u apex bash -lc "cd /var/www/apex-django && uv sync --extra sentry" # 2. Add to /etc/apex-django.env: SENTRY_DSN=https://abc123@oXXXXX.ingest.sentry.io/PROJECT_ID SENTRY_ENV=production # or "demo", "staging", etc. SENTRY_TRACES_SAMPLE=0.05 # 5% perf trace sampling # 3. Restart Daphne sudo systemctl restart apex-django{% endcode %}

When SENTRY_DSN is empty (default), Sentry is skipped entirely — the import is wrapped so the app boots even if the sentry-sdk package isn't installed. Safe to leave unset in dev / private deploys.

Sentry config notes

  • send_default_pii=False — usernames + IPs aren't shipped to Sentry by default. Flip in prod.py if you have explicit user consent.
  • SENTRY_TRACES_SAMPLE — performance trace sample rate (0.0–1.0). On a busy site even 1% can be a lot — start low.
  • SENTRY_RELEASE — bind to a git SHA in CI ($(git rev-parse --short HEAD)) so Sentry tags events with the deploy version.

Logs

Daphne's --access-log - flag writes the access log to stdout, which systemd captures into the journal:

{% code lang="bash" %}sudo journalctl -u apex-django -f # tail live sudo journalctl -u apex-django --since '1 hour ago' | grep WSREJECT{% endcode %}
