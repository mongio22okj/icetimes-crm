Why no React / Vue / Svelte?

Apex is built for product teams that ship features over rebuilds. Server-rendered Django + HTMX + Alpine gets you ~95% of what an SPA gives a typical dashboard, with ~10% of the moving parts: no build step beyond Tailwind, no state synchronization between server and client, no bundle size budget. Each page ships ~30KB of JS total for the entire framework set.

If you're building a real-time collaborative editor, this isn't your template. For ~99% of dashboards, it is.

Does the realtime layer scale beyond a single Daphne process?

Yes — set REDIS_URL in the env and the channel layer transparently upgrades from in-memory to channels-redis. Notification fan-out and presence broadcasts then work across processes. A single Daphne handles thousands of concurrent visitors comfortably; scale horizontally only when you see real load.

Can I run this on Cloudflare Workers / Pages / Vercel?

No — Apex is a real Python app with a real Postgres backing store and WebSockets. None of those platforms run Python long-running processes (Workers is V8-only, Pages is static + Functions which is also V8). You need a Linux VPS or PaaS that runs ASGI.

Cheapest paths: a $6/mo Hetzner / Digital Ocean droplet, Fly.io, or Render. The deploy guide covers the VPS recipe end-to-end.

What about the Cloudflare CDN?

Cloudflare in proxy mode (orange cloud) in front of your origin absolutely works — and is what the live demo uses. Cloudflare gives you the CDN, DDoS shield, and free SSL termination; the origin runs the Django app. The vhost in deploy/nginx/ is pre-configured for this — it trusts CF-Connecting-IP for visitor IP and enforces HTTP→HTTPS redirect.

Why Daphne and not gunicorn / uvicorn?

Daphne is the reference ASGI server for Django Channels — it's the one path the Channels team tests against. Uvicorn works too (and is faster on pure HTTP), but it has subtle WS lifecycle quirks under heavy load that Daphne handles cleanly.

If you don't use Channels at all (no realtime), gunicorn + the WSGI app from apex/wsgi.py is a totally valid swap. You'll lose the /ws/ endpoints and the live notification fan-out.

How do I disable Channels entirely?

  1. Remove daphne and channels from INSTALLED_APPS in apex/settings/base.py.
  2. Remove ASGI_APPLICATION and CHANNEL_LAYERS from base.py.
  3. Switch the systemd unit's ExecStart= from daphne to gunicorn apex.wsgi:application --bind unix:/run/...

You can leave the consumers + dispatcher in place — they just won't have any clients, and the realtime fan-out becomes a no-op (the try/except swallows the missing channel layer).

Can I rip out an app I don't need?

Yes, each apps/* app is self-contained:

  1. Remove from INSTALLED_APPS.
  2. Remove the URL include from apex/urls.py.
  3. Remove the matching NavItem from apps/core/navigation.py.

Apps that other apps depend on: accounts (User model), core (navigation, breadcrumbs, tables), notifications (used by signals in many apps for fan-out). Everything else is removable.

Why don't tests use a real Postgres?

The unit suite uses SQLite for speed (~90s for ~1000 tests). The two Postgres-specific things — JSONField queries and ArrayField — aren't used in the codebase, so SQLite parity holds. If you add Postgres-specific features (e.g. pg_trgm search), you'll want to switch the test DB to Postgres in pyproject.toml's pytest config.
