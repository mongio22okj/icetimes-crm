What's bundled

Apex ships with English (en) and Spanish (es) translations, a header language picker, and gettext_lazy calls across navigation labels and key UI strings. Switching language updates the active locale via Django's set_language view + a session cookie.

Adding a new locale

Add the locale code to LANGUAGES in apex/settings/base.py:

{% code lang="python" %}LANGUAGES = [ ("en", "English"), ("es", "Español"), ("fr", "Français"), # ← new ]{% endcode %}

Then run makemessages to scaffold the .po file:

{% code lang="bash" %}uv run django-admin makemessages -l fr{% endcode %}

Open locale/fr/LC_MESSAGES/django.po, fill in msgstr for each msgid, and compile:

{% code lang="bash" %}uv run django-admin compilemessages{% endcode %}

The header language picker now lists French as an option. Switching activates it everywhere {% verbatim %}{% trans "..." %}{% endverbatim %} or gettext_lazy is used.

Translating new strings in your code

In Python:

{% code lang="python" %}from django.utils.translation import gettext_lazy as _ class MyView(TemplateView): breadcrumb_title = _("Reports") # lazily evaluated at render time{% endcode %}

In templates:

{% code lang="html" %}{% verbatim %}{% load i18n %}

{% trans "Welcome back" %}

{% blocktrans with name=user.first_name %}Hi, {{ name }}!{% endblocktrans %}

{% endverbatim %}{% endcode %}

After adding new trans / _() calls, re-run makemessages -a to pick them up across all locales, then re-translate and recompile.

Per-user language preference

If you want to remember each user's choice across sessions, store it on the User model and set the language in middleware. The current setup uses Django's session-based persistence, which works for anonymous visitors too but resets after session expiry.
